— The business problem
01

A system modelling an org that no longer existed.

The platform’s hiring operations run on an internal admin panel — the system recruiters, account directors, and admins use to manage companies, jobs, and candidates end to end.

The business goal was straightforward: manage end-to-end hiring operations through a system that scales across more roles, more workflows, and rising data volume.

The system had been built for an earlier business model, and as that model evolved — new roles, new responsibilities, new steps — the platform was patched rather than rethought.

So the brief wasn’t “polish the admin panel.” It was: the system models an org that no longer exists — how do we rebuild it to match reality and survive growth?

— Why this hadn’t been solved
02

Internal tools rot quietly.

They get built once, work “well enough,” and then accrete features one request at a time as the company grows. Nobody redesigns them because they technically function — so the cost stays invisible.

But that cost is real. It shows up as a learning curve for every new hire, as coordination tax between teams who verbally negotiate ownership, and as fragmented work spread across disconnected modules. None of it appears on a bug tracker. It just makes the org slower as it scales.

The admin panel didn’t need better buttons — it needed its underlying model rebuilt before the operational complexity it was hiding became the ceiling on hiring growth.

— Strategic frame
03

Three forces, one alignment.

I aligned business, users, and system up front — and set the metrics the redesign had to move.

Out of scope: the candidate-facing marketplace (a separate workstream). This project was strictly the internal operations layer — scoping it tightly is what made a 10-month structural rebuild tractable.

North Star
Steps to complete a core workflow
A scaling system has to make the common path shorter, not just prettier.
Guardrail
Validation integrity must not degrade
Speed can’t come from deleting the checks that keep hiring accountable. This is why I kept a maker–checker step rather than removing approval friction.
Supporting
The signals beneath
  • Role-confusion incidents
  • Cross-module switches per task
  • New-user time-to-proficiency
— Understanding the system
04

Learning how it actually behaved.

I started by learning the system as it really worked, not as it was documented — exploring the admin panel end to end, mapping workflows across the company, job, and candidate systems, identifying gaps between structure and mental models, and running discussions with stakeholders across every role.

The workflows the system was supposed to support: company creation & management → validation & client communication → job creation & assignment → candidate sourcing & screening → feedback-driven hiring progression.

While mapping these, the core problem surfaced: the system didn’t support how users actually operated.

— The three challenges that mattered
05
Challenge 01
Unclear ownership across roles
  • Responsibilities weren’t defined in the system
  • No one was sure who owned each hiring stage
  • Created constant dependency on others and confusion across teams
Challenge 02
High learning curve from unstructured workflows
  • No structure encoded in the system
  • Users navigated on memory and experience
  • New joiners struggled to become productive
Challenge 03
Fragmented workflow across modules
  • A single task forced users to hop between modules
  • Context broke with every switch
  • Time bled on navigation, not decisions

All three trace back to one root: the system structure, not the screens.

“

This was never a usability problem on individual screens. It was a model problem.

— The reframe

Rebuild the model, then the screens.

The system encoded an organisation that no longer existed. Individual screens weren’t broken — but the relationships between them, the role assumptions baked into the architecture, and the workflow sequence were all residue from an earlier era.

So I restated what we were solving:

From: “Fix the admin panel’s usability issues.”
To: “Rebuild the system’s model of the org — roles, workflow stages, and entity relationships — so the interface can finally match how people actually work.”

Every design decision below is a consequence of fixing the model first.

— Design principles
06
01

Encode the org, don’t assume it

Ownership and permissions live in the system, not in people’s heads.
02

Structure the work into stages

A predictable path beats tribal knowledge.
03

One lifecycle, not many modules

Connect entities so context travels with the user.
04

Speed without losing accountability

Remove redundant steps, keep the validation that protects quality.
— The solution
07
A

Role-based visibility & control

Unclear ownership meant no one knew who was responsible at each stage.

Response
  • Role-based access across all user types — each user sees only the data and actions relevant to them
  • Navigation personalised per role — only relevant modules exposed
  • A maker–checker pattern introduced for accountability
  • Admin-level configuration of roles and permissions

Why maker–checker stayed: it adds deliberate friction. I kept it because the guardrail was validation integrity — removing approval would have sped things up while quietly raising error risk in client-facing hiring decisions.

[ Role-based navigation / Super Admin vs restricted view ]
B

Task & process simplification

No encoded structure meant users navigated on memory; new joiners couldn’t become productive.

Response
  • Hiring turned into a structured, stage-based flow: Sourcing → Screening → Validation → Client Sharing
  • Reduced redundant steps and loops in task progression
  • System-triggered notifications to prompt timely action
  • Standardised workflow behaviour across all hiring activities

This is where the ~40% reduction in task steps comes from — and why a new user can now follow the system instead of needing a veteran to explain it.

[ Structured workflow / stage-based task progression ]
C

Unified system architecture & data model

A single task forced users to hop between modules, breaking continuity and bleeding time.

Response
  • Platform reorganised around core entities — companies, jobs, candidates — with explicit relationships forming one connected hiring lifecycle
  • Candidates trackable consistently across every stage
  • Bottlenecks visible because the lifecycle is connected, not siloed
  • Key data layers separated (candidate data vs. user data) to reduce conflicts and improve scalability

Why separate data layers: candidate and user data were tangled in the old system, causing conflicts and blocking clean scaling. Splitting them was an unglamorous structural call, but it’s what lets the system grow without data collisions.

[ Unified lifecycle / company–job–candidate connection ]
— What I deferred
08

The discipline of sequencing.

A 10-month rebuild forced hard calls. Each deferral protected the North Star or the guardrail — that was my test for cutting.

Sequenced
Audit-log & compliance tooling
  • Valuable, but a layer to add after the role model was sound. Sequencing, not omission.
Deferred
Per-field permissions
  • Deferred in favour of role-level access — per-field control added complexity that outran current need.
Constrained
Self-serve role builder
  • Kept role configuration at admin level to avoid permission sprawl before the model had stabilised.
— Impact

What moved.

This is the metric that validates the structural approach — the improvement came from rethinking the model, not optimising individual screens.

— Learnings
10
01

Structure reduces cognitive load

A predictable, staged system lets people follow the path instead of memorising it.
02

Role clarity is critical in enterprise

Encoding ownership in the system — not in people — removes a whole class of confusion and error.
03

Systems should mirror real-world processes

The redesign worked because it matched how teams actually operate, not how the org looked years ago.
04

Connected data enables insight

A unified lifecycle turns scattered records into clear, faster decisions.
What I’d validate next

Does the system teach itself?

  • Whether new-hire time-to-proficiency actually dropped — the clearest proof the system now self-teaches
  • Whether the connected lifecycle surfaces bottlenecks proactively, or just makes them visible after the fact
  • Whether admin-level role config scales, or teams will eventually need safe self-serve
